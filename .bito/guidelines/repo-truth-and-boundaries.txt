Use the repository's written documentation as review context and check whether the change matches the documented intent.

- Start from README.md, ARCHITECTURE.md, AGENTS.md, CONTRIBUTING.md, and docs/ADRs/ for architectural context.
- Check whether code, tests, and documentation tell the same story. Flag mismatches between implementation and documented architecture or ADRs.
- Treat AGENTS.md as the authoritative guide for sharp edges and invariants. If a change violates an invariant documented there, flag it.
- If CI (CircleCI rspec_rubocop job) or another required check already enforces a merge rule, do not ask for duplicate manual checklists.
- Ask for an ADR update when a change is architecture-significant: adding a runtime dependency, changing the node dispatch mechanism, altering XSS protection, or modifying how asset SDK objects are detected.
- The public API surface of this gem is: Renderer.new(mappings = {}), BaseNodeRenderer (for subclassing), and the node type strings in DEFAULT_MAPPINGS. Changes to these require extra scrutiny — they affect all downstream consumers.
- Internal renderer classes (BaseBlockRenderer, BaseInlineRenderer, individual renderers) are implementation details unless they are explicitly subclassed by documented consumer patterns.
